# <a name="head-gaze-and-dwell"></a>Направление головы и остановка

Когда руки заняты инструментами и деталями, жестикулировать может быть сложно или невозможно. При определенных обстоятельствах, например в чрезмерно шумных условиях, на голосовые команды, такие как жесты, нельзя положиться. Кроме того, использование голоса для управления компьютерами не всегда доступно, но оно, безусловно, становится более популярным! "Направление головы и остановка" предлагает наиболее знакомый и простой в освоении механизм для работы в режиме "внимание" и "громкая связь" на HoloLens. Кроме того, опция "Направление головы и остановка" на 100 % надежна, независимо от шумовых помех и проблем с тишиной в рабочей среде.

## <a name="scenarios"></a>Сценарии

Модель "Направление головы и остановка" превосходна в условиях, когда руки человека заняты другими задачами, а системы голосовой связи не на 100 % надежны или недоступны из-за условий окружающей среды или социальных факторов. Хорошим примером является пользователь, носящий HoloLens для получения справочной информации при ремонте двигателя автомобиля. Их руки заняты инструментами или поддержкой своего тела, когда они нагибаются в моторный отсек. Гараж — шумный, с постоянным грохотом и гудением инструментов, что создает помехи голосовым командам. "Направление головы и остановка" позволяет пользователю HoloLens уверенно просматривать свои справочные материалы, не прерывая рабочий процесс.

## <a name="goals"></a>Цели

Создайте механизм для взаимодействия без помощи рук и без использования голоса.

## <a name="design-principles"></a>Принципы проектирования

1. Избегайте "взгляда, как оружие"

    "Направление головы и остановка" требует, чтобы визуальный отзыв был понятным, но слишком перенасыщенный отзыв может вызвать беспокойство. Отзыв должен помочь пользователю узнать, на что он нацелен, но не выбирать цель автоматически по своему усмотрению. Чтение текста, значков и меток требует дополнительного внимания, чтобы предоставить человеку возможность усвоить информацию перед выбором.
    
2. Поиск безопасной скорости
    
    Взаимодействия с остановкой могут иметь разные таймеры, основанные на влиянии навигации. Более часто используемые функции обычно выигрывают от более короткого времени заполнения, в то время как более последовательные функции могут выиграть от более длительного времени заполнения. При использовании способа заполнения для отображения этих таймеров анимационные кривые цвета заполнения могут положительно влиять на ощущение более короткого времени заполнения. Необходимо принять во внимание возможность принятия пользователем решения по переопределению скорости заполнения в пользу быстрой, средней или медленной.
    
3. Скажите "нет-нет" эффекту йо-йо

    Эффект йо-йо — это неудобный шаблон движения головы, который может возникнуть, когда размещение содержимого, контроль за направлением головы и остановкой заставляет людей постоянно смотреть вверх и вниз. Например, навигация по списку с кнопкой "Направление головы и остановка", расположенной внизу, создает замкнутый круг: смотреть вниз, чтобы остановиться, смотреть вверх на результаты, смотреть вниз, чтобы остановиться и т. д. В результате этот шаблон неудобен, и его следует избегать, размещая элементы управления навигацией в централизованном расположении, которое требует меньше движений взад-вперед. Расположение кнопок остановки относительно их воздействия становится важным для комфорта.

## <a name="ux-guidelines-and-best-practices"></a>Руководства и рекомендации по взаимодействию с пользователем

### <a name="target-sizes"></a>Размеры цели
  Для того, чтобы целевые объекты "Направления головы и остановки" были легко доступны, они должны быть достаточно большими, чтобы на них можно было комфортно нацелиться и устойчиво держать голову на цели в течение предписанного времени. Для достижения максимально комфортных ощущений мы рекомендуем минимальный размер цели 2 градуса. 

### <a name="visual-feedback"></a>Визуальная обратная связь

При использовании радиальной заливки для представления таймера остановки начинайте с центра кнопки. Последовательный ответ меньше сбивает с толку, чем все разные направления на разных кнопках. 

  * Это правило может быть нарушено для направленных взаимодействий (например, навигация вверх/вниз/влево/вправо и т. д.). Например, "Руководство по Microsoft Dynamics 365" делает исключение для ДАЛЕЕ/НАЗАД при заливке слева направо.
  * Подумайте о перевертывании радиальной заливки снаружи при выключении кнопки. Ощущение инверсии после нажатия кнопки — хороший визуальный шаблон, который следует поддерживать. 

### <a name="progressive-disclosure"></a>Поэтапное представление информации

Поэтапное представление информации означает только представление того количества деталей, которое уместно на каждом этапе взаимодействия. Для остановки это означает, что целевые объекты остановки обнаруживаются при выделении (например, в элементе управления "Список").

 ### <a name="oversized-targets"></a>Целевые объекты слишком большого размера
Область остановки может быть больше, чем неактивный значок, чтобы проще было использовать, например, кнопку "Назад" в Руководстве по Microsoft Dynamics 365.

### <a name="prevent-flickering-with-delayed-feedback"></a>Предотвращение мерцания с помощью задержки ответа
Используйте короткую задержку перед началом визуального ответа, чтобы избежать мерцания, когда кто-то не заметит целевой объект остановки.
* Для часто взаимодействующих кнопок задержка должна быть очень короткой, чтобы приложение отреагировало.
* Для кнопок, которые нечасто взаимодействуют, целесообразным может быть использование более длительной задержки, чтобы избежать ощущения дергающегося интерфейса.

## <a name="ui-patterns"></a>Шаблоны пользовательского интерфейса

### <a name="high-frequency-buttons"></a>Высокая частота кнопок
![Кнопка "Далее" руководства по Microsoft Dynamics 365](images/GuideNextButton.png "Кнопка \"Далее\" руководства по Microsoft Dynamics 365")<br>
*Кнопки с высокой частотой — это кнопки, которые обычно используются во всем приложении. Хорошим примером являются кнопки Далее и назад в руководствах по Microsoft Dynamics 365.*

Часто использующиеся кнопки должны...
* быть более крупными, чтобы их легче можно было активировать с помощью направления головы;
* быть на уровне глаз для избежания эргономичной перегрузки.

### <a name="low-frequency-buttons"></a>Редко использующиеся кнопки
Редко использующиеся кнопки — это кнопки, не взаимодействующие в приложении регулярно. Наглядным примером может быть кнопка для доступа к меню параметров или кнопка для очистки всей работы.

* Попытайтесь, чтобы на эти кнопки не влияла частая смена направления головы, чтобы избежать случайной активации. 

### <a name="confirmations"></a>Подтверждения
![Диалоговое окно подтверждения в Руководстве Microsoft Dynamics 365](images/GuidesConfirmation.png "Диалоговое окно подтверждения в Руководстве по Microsoft Dynamics 365")

Когда какое-либо действие оказывает существенное влияние, такое как снятие денег, удаление работы или запуск длинного процесса, полезно подтвердить, что пользователь хотел выбрать кнопку. Для пользовательских интерфейсов опций "Направление головы" и "Остановка" есть несколько шаблонов и разработок для диалоговых окон подтверждения.

  * Показать выбранные выделения на главной кнопке.
  * Обнаружить целевой объект остановки одновременно с выбранным выделением.
  * Для дополнительной кнопки показать целевой объект остановки в направлении головы.
        
### <a name="toggle-buttons"></a>Выключатели
Выключатели требуют некоторой детализированной логики для правильной работы. Когда пользователь останавливается на выключателе и активирует его, ему необходимо выйти из кнопки и затем вернуться, чтобы перезапустить логику остановки. Важно, чтобы переключаемые кнопки имели четкое активное или неактивное состояние. 

### <a name="list-views"></a>Представления списка
![Диалоговое окно подтверждения в Руководстве Microsoft Dynamics 365](images/GuidesListView.png "Диалоговое окно подтверждения в Руководстве по Microsoft Dynamics 365")<br>
*Представления списка представляют собой определенную задачу для вдаваясь и ввода в голову. Пользователи должны иметь возможность просматривать содержимое, не тратя на то, чтобы типтое вокруг целевых объектов вдаваясь.*

Несколько советов по представлению списков:
* выделите всю строку, когда используется направление головы, и не останавливайтесь, если только направление головы не находится на целевом объекте остановки;
* показывайте целевой объект остановки только тогда, когда строка выделена, чтобы уменьшить визуальный шум;
* будьте четкими и последовательными с положением целевого объекта остановки;
* не показывайте все целевые объекты остановки сразу, чтобы избежать повторяющегося пользовательского интерфейса;
* повторно используйте один и тот же шаблон как можно чаще, чтобы лучше взаимодействовать с пользователем.
